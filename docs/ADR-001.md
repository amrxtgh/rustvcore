# ADR-001: Learning-First Direction for rustvcore

- Status: Accepted
- Date: 2026-03-06
- Owners: rustvcore maintainers

## Context

`rustvcore` is a personal learning project to understand emulator engineering deeply, not a production emulator.

The current codebase already has:
- A CPU state model (`regs`, `pc`, privilege mode)
- Instruction handler scaffolding for major ISA groups
- A basic RAM and machine composition

Most handlers are intentionally stubbed so implementation can be learned step-by-step.

## Why This Project Was Chosen

- Emulator work forces end-to-end systems thinking (ISA, memory, traps, tooling).
- RISC-V has clean public specifications and a modular extension model.
- Rust gives strong safety guarantees while still exposing low-level control.
- A ground-up implementation gives deeper learning than only reading existing emulator code.

## Decision

The project will optimize for learning clarity first:
- Keep architecture explicit and readable
- Implement instruction logic incrementally
- Document design decisions and roadmap in ADRs
- Prefer correctness and observability before performance optimization

## Alternatives Considered

1. Build a production-focused emulator first
- Rejected because it would prioritize speed/features over understanding.
2. Start directly with advanced extensions (`V`, `B`, peripherals)
- Rejected because without a solid RV32I base, debugging becomes unclear.
3. Focus on JIT early
- Rejected because optimization before correctness reduces learning quality.

## Learning Motive

This project exists to learn:
- How instruction decode/execute really works
- How CPU state evolves per instruction
- How memory, traps, and CSRs interact with execution
- How to grow from minimal ISA support to realistic system behavior

## Goals

- Build a clear and modular RISC-V emulator core in Rust
- Reach stable RV32I behavior first, then grow extension coverage
- Add debugging and tracing tools to support understanding
- Add peripheral and platform features in a controlled roadmap

## Non-Goals (Current Phase)

- Immediate production-grade performance
- Full Linux compatibility from day one
- Implementing all extensions before base correctness is verified

## Roadmap TODO

### Phase 1: Core Correctness

- Add instruction decode representation (`Instruction` enum)
- Add decode + dispatch execution loop
- Implement RV32I core arithmetic/logic/branch/load/store behavior
- Implement memory semantics and sign/zero extension behavior
- Add instruction-level unit tests

### Phase 2: System Semantics

- Add trap/exception handling framework
- Add `ecall`, `ebreak`, and minimal CSR behavior
- Expand integration tests

### Phase 3: Extension Growth

- Implement `M` extension handlers
- Implement `A` extension handlers
- Implement `F` extension handlers (or postpone based on learning priorities)

### Phase 4: Tooling and Observability

- Add instruction tracing/logging
- Add step mode and register dump view
- Add GDB stub integration

### Phase 5: Platform and Performance

- Expand syscall support for Linux userspace programs
- Add VirtIO devices incrementally
- Evaluate JIT only after interpreter correctness is strong

## Future Feature Targets (From User Reference)

The project intends to explore these areas:
- `V` extension: vector/SIMD support
- `B` extension: bit manipulation (`Zba`, `Zbb`, `Zbc`, `Zbs`)
- `Zfinx`: floating-point in integer registers
- GDB stub debugger integration
- Step-by-step tracing and register dump viewer
- Better peripheral support: VirtIO GPU, VirtIO net
- More syscall coverage for Linux userspace
- JIT compilation path
- UI/visualization: web register view, memory viewer, pipeline visualization

## Implementation Order (Planned)

1. RV32I decode/execute correctness
2. Traps + minimal CSR/system behavior
3. M/A/F extensions
4. Debugging tools (trace, step, register dump, GDB stub)
5. Peripheral expansion and syscall coverage
6. Performance and UI features (JIT, visualization)

## Consequences

Positive:
- Easier to reason about and learn from each subsystem
- Better long-term maintainability as complexity grows

Tradeoff:
- Slower feature velocity compared to quick prototype coding
- Performance work is intentionally delayed

## Acceptance Criteria

This ADR is considered active when:
- `README.md` points to this ADR for project direction
- New core milestones align with the roadmap above
- Major scope changes are captured in new ADRs
